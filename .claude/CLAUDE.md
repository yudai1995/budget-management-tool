## 必須プロセス（Hookで強制。違反時はユーザーが検知する）

ユーザーの指示を受け取ったら、以下のフローを順守せよ。特に 1〜3 はコード修正やファイル読み込みの**前に**実行しなければならない：

1. **/branch** — `main` ブランチにいる場合は、必ず作業ブランチを作成せよ。作業内容に基づいた英語 slug を生成すること。
2. **/translate** — 日本語の指示を「ユーザーの意図 → 技術的課題 → 修正対象の具体的ファイル」へ2段階翻訳し、自身の理解を言語化せよ。
3. **/search (影響調査)** — 修正対象のファイルだけでなく、その定義を参照している箇所や、型定義の依存関係を grep 等で調査し、影響範囲を特定せよ。
4. **実装と自己検証** — コード修正後は、必ず `/lint` やビルドコマンドを実行し、構文エラーがないことを確認せよ。可能であれば、関連する既存テストを実行し、デグレがないことを確認すること。

このフローを**スキップしてはならない**。`.claude/settings.json` の Hook でも同様の制約が課されている。

## コミット・プッシュに関する厳格な制約

- **無断コミット・プッシュの禁止**: ユーザーから「コミットして」「プッシュして」という**明示的な指示があるまで、いかなる理由があっても `git commit` および `git push` を実行してはならない。**
- 修正が完了した後は、まず差分を提示し、ユーザーに確認を求めること。

## プロジェクト概要

個人向けの家計管理ツール。支出（Expense）の登録と管理を主目的とするモノリポジトリ構成。

■ **コンセプト**

- 「迷わず、素早く、正確に」：日々の支出をストレスなく記録し、資産状況を透明化する。
- AI駆動開発（Agentic Workflow）に最適化された設計：AIが自律的に、かつ安全にコードを改善できる構造を持つ。

■ **背景・目的**

- 既存の家計簿アプリでは「分類の煩雑さ」や「データの不透明性」が課題。
- 自分の資産データを自分でコントロールし、DDD等の堅牢な設計を通じて、長期的な保守性と拡張性を備えたツールを構築する。
- 支出登録を入り口とし、将来的には収入管理や資産推移の可視化へと繋げる。

## ディレクトリ構造

- `apps/frontend/`: React/Next.js 等を用いたフロントエンド
- `apps/backend/`: Node.js/Express を用いたバックエンド
- `packages/common/`: FE/BE共用の型定義やユーティリティ

## 命名規則（ユビキタス言語）

- **支出**: `Expense` を使用する。
- **収入**: `Income` を使用する。
- **予算/一括呼称**: `Budget` という言葉は、混乱を避けるためビジネスロジック内では原則使用禁止とする。既存の `budget_list` テーブルは `Expense` の永続化先として扱う。
- **ID**: `ulid` を使用する。

## 共通コーディング規約（TypeScript）

- **型定義の徹底**:
    - `any` の使用は原則禁止。やむを得ない場合は理由をコメントし `unknown` を検討する。
    - 型推論を優先するが、関数の引数と戻り値には必ず型注釈を付与する。
- **非同期処理**:
    - `async/await` を使用し、`Promise.then()` は避ける。
    - `try-catch` によるエラーハンドリングを適切に行い、握りつぶさない。
- **マジックナンバーの排除**:
    - 意味のある数値や文字列は定数（`const`）または `as const` オブジェクトとして定義する。

## バックエンド規約（DDD / Onion Architecture）

- **Domain層 (`backend/src/domain/`)**:
    - 外部ライブラリへの依存禁止。値オブジェクト（Value Object）を活用する。
- **Application層 (`backend/src/application/`)**:
    - 1つのユースケースは1つの責務のみを持つ。
- **Infrastructure層 (`backend/src/infrastructure/`)**:
    - 外部接続（DB, API）の詳細はここに閉じ込める。
- **Presentation層 (`backend/src/presentation/`)**:
    - リクエストバリデーションを行い、不正なデータはユースケースに渡さない。

## フロントエンド規約

- **Component設計**:
    - 1つのコンポーネントは 150行以内を目安にする。
    - 副作用（`useEffect`）の利用は最小限にし、イベントハンドラや `useMemo` で処理する。
- **UI/UX**:
    - ローディング状態とエラー状態の表示を必ず考慮する。
    - 破壊的な操作の前には必ず確認（モーダル等）を挟む。

## Git操作のベストプラクティス

- **コミットの粒度**: Atomic Commit（1変更＝1コミット）を徹底。
- **コミットメッセージ**: `type: description` 形式。正確かつ事実に基づき、網羅的に記載する。
    - `type`: `feature`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `chore`
- **ブランチ運用**: `{type}/{description}` 形式。

## 開発・修正時の禁止事項

- **「ついで」の修正禁止**: 指示外の箇所をリファクタリングしてはならない。
- **削除の確認**: コード削除時は影響範囲を報告し、許可を得ること。
- **コメントの維持**: 既存の JSDoc や注釈を勝手に削除しない。

## 技術スタック

- **Runtime**: Node.js / TypeScript
- **Package Manager**: pnpm (Workspaces)

## AIへの行動制約

- **正確な情報源の提示**: 根拠を明示すること。
- **簡潔な回答**: 結論から述べ、冗長な挨拶を省く。
- **過度な比喩の禁止**: 情緒的な表現を避ける。
- **不明確な事項の申告**: 不確実な場合はその事実を伝える。
